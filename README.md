The app currently uses linear regression and random forest as working placeholders, but is designed so that new model types can be added with minimal changes (see Adding New Models below).

| File | Role | When to touch it |
|------------------------|------------------------|------------------------|
| `app.R` | The Shiny app itself | When changing the UI, adding inputs, or changing how predictions are displayed |
| `background code.R` | Offline training script | When retraining models (new data, new model types, changed transformations) |

There are 31 `mod_*` pairs and 31 `meta_*` files, one for each non-empty subset of the five predictor groups.

## How the app works

The user can provide any combination of five inputs: number of vials, transfer volume, difficulty, batch status, and facility type. Rather than running through complex conditional logic to handle every possible input combination at runtime, `background code.R` pre-trains a model for all 31 possible non-empty subsets of those five inputs. The models are saved as `.rds` files with names that encode which predictors they use.

When a user submits inputs, the app:\
1. Checks which inputs were provided.\
2. Constructs a **combo_tag** (a string like `Vials_Transfer_volume_Batch`) that names that exact combination.\
3. Reads the corresponding `meta_<combo_tag>.rds` to find which model type (linear regression or random forest) had the lower test RMSE for that combination.\
4. Loads that model file, applies the same transformations used during training, predicts, and back-transforms from log scale to minutes.

This repository of pre-trained models that the app can recall as needed keeps the application lightweight and easy to deploy to the web.

### Canonical tagging system

The combo_tag is built by joining group names with underscores in a fixed canonical order:

```         
Vials ⟶ Transfer_volume ⟶ Difficulty ⟶ Batch ⟶ Beds
```

This order is hardcoded in two places that must always agree:

- The `provided <- c()` block in `predict_time()` in `app.R`, and

- The `predictor_groups` list definition in `background code.R`

**If you add a new predictor group: update both places and keep the order consistent.** A mismatch means the app will generate a tag that doesn't match any saved file.

### Log transformation

All continuous variables were log-transformed before modeling because compounding time is strongly right-skewed. The log transform stabilizes variance, improves linearity, reduces the influence of extreme values, and keeps predictions positive. It also aligns with the multiplicative nature of compounding workload and improves performance for both linear regression and random forest models.

`Compound_time` ⟶ `log(Compound_time)` (response)

`Vials` ⟶ `log(Vials)`

`Transfer_volume` ⟶ `log(Transfer_volume + 0.1)` (the +0.1 prevents `log(0)` for zero-volume transfers as a safety catch, but there is input validation in place to prevent users from entering a 0mL transfer volume)

Therefore, model predictions are on the log scale. `predict_time()` calls `exp()` to convert back to minutes before displaying the result. If you change any transformation in `background code.R`, you **must** update the corresponding line in `app.R`'s `predict_time()` function or predictions will be wildly wrong.

## How to set up and run locally

1.  Launch RStudio.

2.  Install the required packages:

    ``` r
    install.packages(c("shiny", "randomForest", "ggplot2", "dplyr", "rsconnect", "tidyverse"))
    ```

3.  Clone or copy the app directory to your computer.

4.  Open `background code.R`. Update the two hardcoded paths:

    - path inside `readRDS()` in Section 1 to wherever your local copy of the dataset lives
    - `model_out_dir` path in Section 6 to wherever your local copy of the app directory lives

5.  Run `background code.R` from top to bottom. This takes a few minutes and generates all 62 model files and 31 metadata files. If you update anything in `background code.R`, you will need to re-run it in order for your changes to take effect in `app.R`. I highly recommend deleting every `.rds` file from your app directory folder before each re-run to prevent any overwriting issues and outdated models from cluttering up the directory.

6.  Open `app.R` in RStudio and click **Run App**, or run `shiny::runApp("path/to/app/directory")` from the console.

## How to deploy to shinyapps.io

Deployment makes the app accessible via a public URL without the user needing R installed.

1.  Create a free account at [shinyapps.io](https://www.shinyapps.io) if you don't have one.

2.  In RStudio, install the rsconnect package:

    ``` r
    install.packages("rsconnect")
    ```

3.  Configure your credentials. Find your token and secret in the shinyapps.io dashboard under Account ⟶ Tokens:

    ``` r
    rsconnect::setAccountInfo(name   = "your-account-name",
                              token  = "your-token",
                              secret = "your-secret")
    ```

    This only needs to be done once to link your code and directory to the URL.

4.  Deploy by running this from the R console:

    ``` r
    rsconnect::deployApp("path/to/your/app/directory")
    ```

    This uploads `app.R` and all `.rds` files in the directory to the web and can take a few minutes. Make sure you have run `background code.R` first so the model files exist. Also, make sure `app.R` and the `.rds` files are the *only* things stored in the directory folder. After the first deployment, an "rsconnect" folder will appear in the directory – don't touch it.

5.  After deployment finishes, shinyapps.io will print the live URL. Test it in a browser. You might wish to occasionally try it in a few different browsers to make sure your app code is still universally compatible.

**Important:** If you update the models or make changes to the app, you must re-deploy for the changes to take effect on the web version. You do not need to re-configure credentials again after the initial setup.

## Developer Mode

The app has a "Developer Mode" checkbox at the bottom of the input panel. When checked, the right-hand column shows:

- **combo_tag** – the predictor combination string that was used
- **newdata structure** – a `str()` view of the single-row data frame sent to `predict()` (useful for confirming transformations are correct)
- **Selected model + test RMSE** – which model type was chosen and its held-out RMSE in minutes
- **Model comparison chart** – a bar chart showing the prediction and RMSE error bar for every model saved for this combination, side-by-side
- **Model comparison table** – the same information in tabular form

This panel is intended to be a helpful diagnostic tool for future developers, not clinical users. The error bars in the chart show prediction ± test RMSE. This is a rough indication of model uncertainty and not a formal confidence interval.

## Adding new models

The app is designed to accept new model types without requiring extensive changes to `app.R`. The steps are:

1.  **In `background code.R`, Section 7:** fit your new model inside the loop, following the pattern used for `lm_fit` and `rf_fit`. Be sure to do the same under the "evaluate on test data" and "select best model by test RSME" sections
2.  **Re-run `background code.R`** to regenerate all model files and metadata.
3.  **In `app.R`**, add your new model to `model_label` before the final `list()` return at the end of `predict_time()`.
4.  **Re-deploy** the app.

The `meta_<combo_tag>.rds` files will automatically record which model type won for each combination, and `app.R` will select it without any code changes. The Developer Mode comparison table will also automatically include your new model type.

------------------------------------------------------------------------

## Adding new predictor inputs

Adding additional inputs requires changes in three places:

1.  **`background code.R`, Section 4:** add a new entry to `predictor_groups` and re-run the script to train models for all new combinations
2.  **`app.R`, `predict_time()`:** add the new predictor to the `provided <- c()` block and build the corresponding column in `newrow`, applying the same transformation used at training time.
3.  **`app.R`, UI section:** add a new input widget (`numericInput`, `selectInput`, etc.) for the new predictor.

Remember that the canonical order in `provided <- c()` and in `predictor_groups` must match!

## Understanding the data

The dataset comes from an IV workflow management system (IV WFMS) used across four hospitals in a health system. The raw dataset contained \~395,000 doses. After Changfei's cleaning (exclusion of pediatric doses, interrupted doses, non-standard procedures, etc), \~114,000 doses remained.

The five predictor variables used in this app were identified by Changfei via stepwise regression as the most useful subset:

| Variable | Type | Description |
|------------------------|------------------------|------------------------|
| Vials | Continuous | Number of product vials per dose |
| Transfer_volume | Continuous | Total mL transferred per dose |
| Difficulty | Ordinal (1–5) | Container type difficulty: 1=syringe, 5=on-Q pump |
| Batch | Binary | 0=single dose, 1=batch dose |
| Beds | Categorical | Hospital size category (100–1000 beds, Cancer, Outpatient) |

For full data dictionary details, see Changfei's cleaning code and report.

## Known limitations

- **Data scope:** The training data is from one health system over one quarter. Predictions may be less accurate for systems with very different compounding workflows.
- **Placeholder models:** lm and random forest are intended as placeholders. Exploring more sophisticated modeling approaches is encouraged.
- **No prediction intervals:** The app displays a point estimate only. Displaying a formal uncertainty interval would require additional work.
